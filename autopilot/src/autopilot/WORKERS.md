# Workers

Workers are the executable units of an agent. They are Python functions
decorated with `@tool` from the Agentspan SDK that perform real work: calling
APIs, running computations, processing data, generating files. When no builtin
integration covers a capability the user needs, the Claw orchestrator writes a
custom worker, validates it, installs its dependencies, and wires it into the
agent.

Workers are stored as `.py` files in:

```
~/.agentspan/autopilot/agents/<agent_name>/workers/
```

Each file contains exactly one `@tool`-decorated function. The agent loader
(`autopilot.loader`) imports these files at deploy time, finds the decorated
functions via the `_tool_def` attribute, and hands them to the `Agent` instance.

---

## Language and Constraints

- **Python only.** Workers are always Python 3.10+.
- **`@tool` decorator is mandatory.** Every worker function must be decorated
  with `@tool` from `agentspan.agents`. The decorator generates the JSON schema
  the LLM sees when deciding which tool to call.
- **Type hints on all parameters.** The `@tool` decorator reads annotations to
  build the schema. Missing type hints cause a validation gate failure.
- **Return type is always `str`.** For structured data, use `json.dumps()` or
  `str()`. The validation gate calls the function and asserts `isinstance(result, str)`.
- **Module-level imports are limited to `from agentspan.agents import tool`.**
  All other imports (requests, json, os, etc.) go inside the function body,
  inside the try block. Worker files are loaded via `importlib` in forked
  subprocesses; module-level imports of third-party packages fail if the package
  is installed in the parent process but not the subprocess environment.

## Worker Generation Process

When the orchestrator determines a capability gap (no builtin integration
covers what the user asked for), it calls `generate_worker`. Here is the
full flow:

1. **Capability gap identified.** The orchestrator decomposes the user's
   request into capabilities and checks the integration registry. If no
   builtin covers a capability, a custom worker is needed.

2. **`generate_worker` called** with these parameters:
   - `agent_name` -- which agent gets the worker
   - `tool_name` -- snake_case Python identifier for the function
   - `description` -- what the tool does (becomes the docstring the LLM reads)
   - `parameters` -- comma-separated typed params, e.g. `"limit: int = 1000"`
   - `implementation` -- the actual Python code for the function body
   - `dependencies` -- comma-separated pip package names, e.g. `"requests,beautifulsoup4"`

3. **Directory setup.** The agent directory and `workers/` subdirectory are
   created if they don't exist.

4. **Tool name validation.** Must be a valid Python identifier
   (`str.isidentifier()`).

5. **Implementation wrapping.** The implementation code is wrapped in
   `try/except` so the worker never crashes:
   - If the implementation starts with `def `, the body is auto-extracted
     (the LLM sometimes writes a full function definition instead of just
     the body).
   - If no `return` statement is present, one is auto-added.
   - The entire body is indented inside a try block, with an except clause
     that returns `f"Error in {tool_name}: {_err}"`.

6. **Dependencies recorded.** If `dependencies` is non-empty, the packages
   are merged into `workers/requirements.txt` (existing entries preserved,
   new entries added, sorted alphabetically).

7. **Dependencies installed.** `uv pip install` is attempted first (uv-managed
   venvs don't have pip). Falls back to `python -m pip install` if uv is not
   available.

8. **AST validation.** The generated code is parsed with `ast.parse()`. If it
   has a syntax error, the error is returned immediately.

9. **File written.** The worker is saved to
   `workers/<tool_name>.py`.

10. **agent.yaml updated.** The tool name is appended to the `tools` list in
    the agent's `agent.yaml` so the loader picks it up at deploy time.

(line ~1299).

## Worker Anatomy

A complete, well-formed worker file:

```python
"""Worker: compute_primes -- Generate prime numbers using Sieve of Eratosthenes

Auto-generated by Agentspan Claw orchestrator. Workers never crash --
all exceptions are caught and returned as error strings so the agent
can reason about them and retry.
"""

from __future__ import annotations

from agentspan.agents import tool


@tool
def compute_primes(limit: int = 1000) -> str:
    """Generate all prime numbers up to the given limit."""
    try:
        sieve = [True] * (limit + 1)
        sieve[0] = sieve[1] = False
        for i in range(2, int(limit**0.5) + 1):
            if sieve[i]:
                for j in range(i * i, limit + 1, i):
                    sieve[j] = False
        primes = [i for i, is_prime in enumerate(sieve) if is_prime]
        return str(primes)
    except Exception as _err:
        return f"Error in compute_primes: {_err}"
```

### Structure breakdown

| Section | Purpose |
|---------|---------|
| Module docstring | `Worker: <name> -- <description>`, auto-generated notice |
| `from __future__ import annotations` | Forward-reference support |
| `from agentspan.agents import tool` | The only allowed module-level import |
| `@tool` decorator | Registers the function as an LLM-callable tool |
| Type-hinted parameters with defaults | Generates the JSON schema; defaults used in test-runs |
| Docstring on function | Shown to the LLM as the tool description |
| `try:` block | All implementation code lives here |
| `except Exception as _err:` | Catches everything, returns error string |

## Principles

### No Fakes

Every worker MUST perform real work. No simulations, no "in production this
would...", no `return f"Text message sent to {phone}: {message}"`. If the
action cannot be performed (missing credentials, missing API), the worker must
return an honest error -- not a simulated success. The orchestrator must use
existing builtin integrations before writing custom workers.

### Never Crash

All implementation code is wrapped in `try/except`. Workers return error
strings, never raise exceptions. This lets the agent reason about what went
wrong and retry with different parameters or a different approach.

```python
try:
    # real work here
    return result_string
except Exception as _err:
    return f"Error in my_tool: {_err}"
```

### Self-Contained Imports

All imports (except `from agentspan.agents import tool`) must be INSIDE the
function body, inside the try block. This is not a style preference -- it is
a technical requirement. Worker files are loaded via `importlib.util` in
subprocess forks. Module-level imports of third-party packages that are only
installed in the parent environment will cause `ModuleNotFoundError` in the
subprocess.

```python
@tool
def fetch_page(url: str = "https://example.com") -> str:
    """Fetch a web page and return its text content."""
    try:
        import requests               # <-- inside the function
        from bs4 import BeautifulSoup  # <-- inside the function
        resp = requests.get(url, timeout=10)
        resp.raise_for_status()
        soup = BeautifulSoup(resp.text, "html.parser")
        return soup.get_text(separator="\n", strip=True)[:5000]
    except Exception as _err:
        return f"Error in fetch_page: {_err}"
```

### Declare Dependencies

Any pip package used must be listed in the `dependencies` parameter when
calling `generate_worker`. These are written to `workers/requirements.txt`
and installed via `uv pip install` (or `pip install` fallback) both at
generation time and at deploy time.

### Return Strings

Workers always return `str`. The validation gate calls the function with
default arguments and asserts `isinstance(result, str)`. For structured data:

```python
import json
return json.dumps({"primes": primes, "count": len(primes)})
```

### Type Hints

All parameters must have type hints. The `@tool` decorator reads them to
generate the JSON schema the LLM uses to decide what arguments to pass.
Missing type hints cause a validation gate failure.

### One Capability Per Worker

Each worker does ONE thing. If an agent needs to search the web AND process
images, it has two workers: `search_web.py` and `process_image.py`. This
keeps workers focused, testable, and independently replaceable.

## Validation Gates

Before an agent is deployed, its workers pass through a series of validation
gates. These gates are `@tool`-decorated functions themselves, so the
orchestrator can call them and react to failures.

### Gate: validate_code

Static analysis of every `.py` file in `workers/`:

| Check | What it does |
|-------|-------------|
| AST parse | `ast.parse(source)` -- catches syntax errors |
| `@tool` decorator | Walks the AST looking for at least one function with `@tool` |
| Type hints | Every parameter on `@tool` functions must have an annotation |
| Security scan | Regex patterns flag dangerous calls |

**Security patterns flagged:**

- `os.system(` -- shell injection risk
- `eval(` -- arbitrary code execution
- `exec(` -- arbitrary code execution
- `subprocess.call(` -- uncontrolled subprocess
- `subprocess.Popen(` -- uncontrolled subprocess
- `__import__(` -- dynamic import bypass

If the agent only uses builtin integrations (no custom workers), this gate
returns PASS with a note rather than failing on missing `.py` files.

### Gate: validate_worker_execution

Runtime validation -- actually runs each worker in a subprocess:

1. Install dependencies from `requirements.txt` (if present).
2. For each `.py` file in `workers/`:
   - Import the module via `importlib.util.spec_from_file_location`.
   - Find functions with `_tool_def` attribute (set by `@tool`).
   - Build default arguments: use `param.default` if available, otherwise
     empty string for `str`, `0` for `int`, `0.0` for `float`, `False` for
     `bool`.
   - Call the function with those arguments.
   - Assert the return value is a `str`.
3. If any worker fails to import, raises an exception, or returns a non-string,
   the gate reports FAIL with the specific error.

**Timeout:** 10 seconds per worker. Workers that hang are killed and reported.

## Deploy Pipeline

When an agent is deployed, the subprocess entry-point
(`_deploy_runner.py`) runs three steps:

1. **Install dependencies.** Reads `workers/requirements.txt` and runs
   `uv pip install` (preferred) or `python -m pip install` (fallback).
   If installation fails, the deploy aborts.

2. **Compile check.** Every `.py` file in `workers/` is passed through
   `compile()` to catch syntax errors before loading.

3. **Load and run.** The `loader.load_agent()` function reads `agent.yaml`,
   resolves builtin tool references via the registry, imports worker modules
   via `importlib.util`, collects `@tool` functions, and constructs an `Agent`
   instance. The agent is then started via `AgentRuntime`.

### Source

`autopilot/src/autopilot/orchestrator/_deploy_runner.py`

## Dependency Management

### requirements.txt

Each agent's `workers/` directory has its own `requirements.txt`. When
`generate_worker` is called with dependencies, they are merged into this file
(existing entries preserved, duplicates removed, sorted alphabetically).

### Installation strategy

```
uv pip install --quiet <packages>    # preferred (fast, works in uv venvs)
python -m pip install --quiet <packages>  # fallback
```

`uv` is preferred because uv-managed virtual environments do not have pip
installed. The system tries `uv` first (via `shutil.which("uv")`) and falls
back to `python -m pip` if uv is not found.

### Installation timing

Dependencies are installed at two points:
1. **Generation time** -- immediately after `generate_worker` writes the file.
2. **Deploy time** -- the deploy subprocess installs from `requirements.txt`
   before importing any worker modules.

### Package name vs. import name

Some pip packages have different import names. Workers must import the correct
module name, while `requirements.txt` uses the pip package name:

| pip package | import name |
|-------------|-------------|
| `beautifulsoup4` | `from bs4 import BeautifulSoup` |
| `pillow` | `from PIL import Image` |
| `python-dateutil` | `import dateutil` |
| `scikit-learn` | `import sklearn` |
| `opencv-python` | `import cv2` |

## Common Patterns

### HTTP API call

```python
@tool
def get_weather(city: str = "San Francisco") -> str:
    """Get current weather for a city using wttr.in."""
    try:
        import requests
        resp = requests.get(f"https://wttr.in/{city}?format=j1", timeout=10)
        resp.raise_for_status()
        data = resp.json()
        current = data["current_condition"][0]
        return (
            f"Weather in {city}: {current['weatherDesc'][0]['value']}, "
            f"{current['temp_F']}F, humidity {current['humidity']}%"
        )
    except Exception as _err:
        return f"Error in get_weather: {_err}"
```

**Dependencies:** `requests`

### Data processing

```python
@tool
def analyze_csv(csv_url: str = "") -> str:
    """Download a CSV and return summary statistics."""
    try:
        import pandas as pd
        df = pd.read_csv(csv_url)
        summary = df.describe().to_string()
        return f"Shape: {df.shape}\n\n{summary}"
    except Exception as _err:
        return f"Error in analyze_csv: {_err}"
```

**Dependencies:** `pandas`

### File generation

```python
@tool
def generate_qr(data: str = "https://agentspan.dev", filename: str = "qr.png") -> str:
    """Generate a QR code image and save it to disk."""
    try:
        import qrcode
        img = qrcode.make(data)
        img.save(filename)
        return f"QR code saved to {filename}"
    except Exception as _err:
        return f"Error in generate_qr: {_err}"
```

**Dependencies:** `qrcode,pillow`

### Web scraping

```python
@tool
def scrape_headlines(url: str = "https://news.ycombinator.com") -> str:
    """Scrape top headlines from a web page."""
    try:
        import requests
        from bs4 import BeautifulSoup
        resp = requests.get(url, timeout=10)
        resp.raise_for_status()
        soup = BeautifulSoup(resp.text, "html.parser")
        titles = [a.get_text() for a in soup.select(".titleline > a")][:15]
        import json
        return json.dumps(titles, indent=2)
    except Exception as _err:
        return f"Error in scrape_headlines: {_err}"
```

**Dependencies:** `requests,beautifulsoup4`

### Pure computation (no dependencies)

```python
@tool
def fibonacci(n: int = 20) -> str:
    """Compute the first N Fibonacci numbers."""
    try:
        fibs = [0, 1]
        for _ in range(2, n):
            fibs.append(fibs[-1] + fibs[-2])
        return str(fibs[:n])
    except Exception as _err:
        return f"Error in fibonacci: {_err}"
```

**Dependencies:** none

## Anti-Patterns

### Module-level imports for third-party packages

```python
# WRONG -- will fail in forked subprocesses
import requests
from bs4 import BeautifulSoup

@tool
def fetch(url: str = "") -> str:
    ...
```

Move all third-party imports inside the function body.

### Returning None

```python
# WRONG -- gate checks isinstance(result, str)
@tool
def do_thing(x: str = "") -> str:
    try:
        process(x)
        # forgot to return!
    except Exception as _err:
        return f"Error: {_err}"
```

Always return a string from every code path. `generate_worker` auto-adds a
return statement if one is missing, but hand-written workers must be explicit.

### Full function definitions as implementation

When calling `generate_worker`, the `implementation` parameter should be the
function body only, not a full `def` statement:

```python
# WRONG -- generate_worker has to detect and strip this
implementation = "def compute(x):\n    return str(x * 2)"

# RIGHT -- just the body
implementation = "result = input_data * 2\nreturn str(result)"
```

`generate_worker` handles the wrong case (it detects `def ` and extracts the
body), but passing just the body avoids an unnecessary fixup step.

### Simulated or fake results

```python
# WRONG -- violates Worker Integrity
@tool
def send_sms(phone: str = "", message: str = "") -> str:
    try:
        return f"SMS sent to {phone}: {message}"  # <-- nothing was sent
    except Exception as _err:
        return f"Error: {_err}"
```

If the worker cannot perform the real action, return an honest error
explaining what is missing (credentials, API access, etc.).

### Using eval/exec/os.system

```python
# WRONG -- flagged by security scan
@tool
def run_code(code: str = "") -> str:
    try:
        return str(eval(code))  # security gate FAIL
    except Exception as _err:
        return f"Error: {_err}"
```

These patterns are caught by regex in `validate_code` and cause a gate failure.

### Hardcoded credentials

```python
# WRONG -- secrets in code
api_key = "sk-abc123..."

# RIGHT -- read from environment
api_key = os.environ.get("MY_API_KEY", "")
if not api_key:
    return "Error: MY_API_KEY not set in environment"
```

Credentials are managed via `agent.yaml`'s `credentials` section and injected
as environment variables at runtime.

## How Workers Are Loaded

The loader (`autopilot/src/autopilot/loader.py`) follows this sequence:

1. Read `agent.yaml` and parse the `tools` list.
2. Separate `builtin:<integration>` references from bare tool names.
3. Resolve builtin references via the integration registry.
4. For bare tool names, check if they match a known integration name (the LLM
   often omits the `builtin:` prefix). If so, resolve via registry.
5. Otherwise, look for `workers/<tool_name>.py`. Import via
   `importlib.util.spec_from_file_location` with a unique module name
   (`_autopilot_worker_<agent>_<tool>`) to avoid collisions.
6. Scan the imported module for objects with `_tool_def` attribute (set by
   `@tool`).
7. Collect all found tool functions and pass them to the `Agent` constructor.
