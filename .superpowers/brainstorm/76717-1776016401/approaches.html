Agentspan Claw — Architecture Approaches

Three approaches for how the product layer sits on top of Agentspan

A

Orchestrator Agent Pattern

A single "Claw Agent" that creates and manages other agents.

The chat interface talks to one meta-agent. When the user says "scan my emails daily," the orchestrator agent uses Agentspan's SDK as tools to: expand the prompt, create the agent definition, write worker code, compile and deploy. The orchestrator itself is an Agentspan agent with tools like create_agent, deploy_agent, schedule_agent, signal_agent.

Pros

  • Elegant — it's agents all the way down
  • The orchestrator can reason about agent design
  • Prompt expansion is just the orchestrator thinking
  • Minimal new infrastructure — it's just another Agentspan agent

Cons

  • Meta-agent reliability — if it hallucinates, it creates broken agents
  • Harder to make deterministic (agent creation should be reliable)
  • LLM cost for every management action
B

Structured Pipeline Pattern

A deterministic pipeline with LLM only for prompt expansion.

The system has distinct phases: (1) Chat UI collects user intent, (2) LLM expands prompt into structured agent spec via templates, (3) Code generator produces agent definition + workers from spec, (4) Agentspan SDK compiles and deploys. Each phase has validation. The LLM is constrained to specific steps, not free-form reasoning.

Pros

  • Predictable — each step is validated before the next
  • Cheaper — LLM only used for expansion, not management
  • Easier to debug — clear pipeline stages
  • Templates ensure correct agent structure

Cons

  • Less flexible — novel agent patterns need new templates
  • More code to maintain (pipeline stages, templates, validators)
  • Can't handle ambiguity as gracefully
C

Hybrid: Orchestrator + Guardrails

An orchestrator agent with structured validation at each step.

Combines A and B. A Claw orchestrator agent handles the chat and reasoning, but agent creation goes through a validated pipeline: (1) Orchestrator expands prompt → validated against schema, (2) Orchestrator generates agent spec → validated for completeness, (3) Code generation for workers → validated/tested, (4) Deploy via SDK. The orchestrator is free to reason, but its outputs are guardrailed at each stage. For management (signal, schedule, monitor), the orchestrator has deterministic tools.

Pros

  • Best of both — flexible reasoning with reliable outputs
  • Guardrails catch hallucinated agent specs before deployment
  • Management actions are deterministic tools (no LLM in the loop)
  • Agentspan's own guardrail system enforces the validation
  • Natural fit — Agentspan already supports guardrails natively

Cons

  • More upfront design to define guardrail boundaries
  • Slightly more complex than pure A or pure B

Recommendation: C — Hybrid

It leverages Agentspan's guardrails natively, keeps the orchestrator flexible for novel requests, and ensures agent creation is reliable. The orchestrator agent is itself an Agentspan agent — so you're dogfooding the platform. Management tools (schedule, signal, list, stop) are deterministic — no LLM needed to run agentspan schedule set.
