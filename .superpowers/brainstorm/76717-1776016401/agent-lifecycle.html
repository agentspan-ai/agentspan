Section 5: Agent Lifecycle & Scheduling

How agents are born, run, evolve, and die — plus the three trigger models

Agent States

Lifecycle State Machine
DRAFT → DEPLOYING → ACTIVE → ARCHIVED
ACTIVE ↔ PAUSED     ACTIVE ↔ WAITING
Any state → ERROR → auto-retry or → PAUSED
DRAFT Agent created locally, not yet deployed. Missing credentials or user hasn't confirmed.
DEPLOYING Compiling on server, registering workers, setting up schedule. Guardrails running.
ACTIVE Deployed and running (or waiting for next trigger). Healthy.
PAUSED User-paused or auto-paused after errors. Schedule suspended. Can resume.
WAITING Agent is mid-execution and needs user input (HITL). Shows in notifications.
ERROR Execution failed. Auto-retries up to 3x, then pauses and notifies user.
ARCHIVED User deactivated. Local folder retained. Server-side schedule removed. Can reactivate.

Three Trigger Models

Scheduled (Cron)

schedule: "0 8 * * *"
# every day at 8am

Server-side cron via Agentspan scheduling API. Agent starts fresh each trigger. Best for: daily summaries, periodic reviews, batch processing.

Event-Driven

trigger: webhook
source: gmail_push_notification

Server receives webhook/event → starts agent execution. Agent processes the event payload. Best for: react to new emails, file uploads, Slack messages.

Always-On (Daemon)

trigger: daemon
poll_interval: 60s

Long-running agent using wait_for_message pattern from 82b TUI. Stays alive, polls data sources, acts when conditions met. Best for: monitoring, live dashboards, chat bots.

Signals — Mid-Execution Control

Signals flow
# From chat thread
You: also skip newsletters from now on
→ Orchestrator sends signal to email-morning-summary agent
→ runtime.signal(execution_id, "skip newsletters in future runs")
→ Agent sees [SIGNALS] skip newsletters [/SIGNALS] on next iteration

# From TUI
/signal email-morning-summary "include calendar events too"

# From dashboard action button
[signal agent] → text input → POST /api/agent/{eid}/signal

# Persistent vs transient
Persistent signal: modifies agent behavior going forward (updates expanded_prompt.md)
Transient signal: affects current/next run only (injected via runtime.signal)

Key distinction: When a user says "also do X from now on" in the chat, the orchestrator recognizes this as a permanent change and updates both the agent's expanded_prompt.md and sends a transient signal for the current run. When they say "skip X this time," it's just a transient signal.
