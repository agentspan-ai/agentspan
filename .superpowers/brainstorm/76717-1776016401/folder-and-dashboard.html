Polling Architecture

Local UI → Remote Server (no server push)
Dashboard poll (every 30s):
  GET /api/agent/executions?status=RUNNING,PAUSED,COMPLETED&since={lastPoll}
  → Returns list of agent statuses + latest outputs

Active chat stream (while chat thread is open):
  GET /api/agent/stream/{executionId} (SSE — client-initiated, long-lived)
  → Real-time events while user is watching

Notification fetch (on dashboard load + poll):
  GET /api/agent/executions?hasOutput=true&since={lastSeen}
  → Agent outputs the user hasn't seen yet

Actions (user-initiated):
  POST /api/agent/{executionId}/signal — send signal to running agent
  POST /api/agent/{executionId}/approve — respond to HITL pause
  POST /api/agent/{executionId}/cancel — stop agent

Key constraint: All data flows from UI → server. The server never pushes to the UI. SSE is client-initiated (browser/TUI opens the connection). Dashboard uses periodic polling for status updates. Notification "unread" state is tracked locally in ~/.autopilot/config.yaml (last-seen timestamps per agent).
